## Summary posture (the good news first)

The codebase is in **strong** shape on the dimensions that were recently invested in:
- **PHPStan level 10 with a baseline now driven to 0** (PR #221) — type-correctness is enforced
  on all new code; the build fails on any new error.
- **God-classes decomposed** — every former 600–900 LOC service/repo is now split into
  unit-tested collaborators; only the documented deferred tail remains.
- **Full gauntlet gating merges** — PHPStan + cs-fixer + Rector + ESLint + 3 PHPUnit suites +
  Qodana, with a baseline ratchet.
- **No hardcoded secrets, no disabled TLS** found in the audited addons.
- **SQL access is parameterized** — the one dynamic-query builder (`DatabaseIterator`) was
  verified to build its `WHERE` from literal placeholders only; values are bound, not interpolated.

The findings below are therefore mostly *hardening* and *blind-spot* items, not active fires.

## Findings

### HIGH

**H1 — Tooling blind spot: the largest, most security-relevant files get no style/import/lint gate.**
`php-cs-fixer`'s finder deliberately excludes `controllers/`, `hooks/`, `functions/`, and the CI
`PHP Lint` job (`.github/workflows/ci.yml:267-270`) runs `php -l` on **`src/` only**. So the big
procedural files — `sphinx func.php` (1223 LOC), `novoton functions/hotels.php` (822),
`travel_core functions/hotels.php` (927), the backend controllers (700+ LOC) — receive **no
auto-formatting, no unused-import cleanup, and no syntax lint** in CI. PHPStan L10 *does* analyze
them (so type/syntax bugs are still caught), but dead imports and style drift accumulate exactly
in the hardest-to-review files (this is why the Qodana pass had to remove dead imports there by
hand). *Fix:* add a second `php-cs-fixer` finder/config covering procedural dirs with a
conservative ruleset (at minimum `no_unused_imports`, `no_whitespace_*`), and extend the
`PHP Lint` job's `find` to include `controllers functions hooks`.

**H2 — [verify] CSRF protection on state-changing frontend endpoints is implicit.**
Only **4 of 46** controller files call `fn_csrf_validate_request`; the booking-mutating ones —
`controllers/frontend/novoton_booking.php`, `update_booking.php`, `novoton_booking/add_to_cart.php`,
`sphinx_holidays/controllers/frontend/sphinx_booking.php` — do **not**. CS-Cart's dispatcher
globally CSRF-protects standard **POST** form submissions, so the common path is likely covered —
but any **GET-triggered mutation** or **custom AJAX handler** that bypasses the standard POST flow
would be exposed. *Fix:* enumerate every state-changing action in these controllers, confirm it is
POST and flows through CS-Cart's central check; add an explicit `fn_csrf_validate_request()` to any
GET/AJAX mutation. (Severity is High pending that enumeration; it may resolve to Low if all are
POST.)

### MEDIUM

**M1 — [verify] Transaction rollback coverage across 9 `START TRANSACTION` sites.**
9 manual transactions exist:
`BookingRepository.php:248/273/343`, `BookingSubmissionService.php:126`,
`HotelRepository.php:171`, `FacilityRepository.php:184`,
`BatchedHotelInfoSyncV2.php:405`, `sphinx SphinxBookingRepository.php:315`,
`DestinationWhitelistRepository.php:131`. The novoton `BookingRepository`/`BookingSubmissionService`
ones were confirmed to wrap in `try/catch` with `ROLLBACK` on every throw path. *Fix:* verify the
remaining five (sphinx + travel-core + the two sync/facility repos) guarantee `COMMIT` **or**
`ROLLBACK` on **every** return/exception path — a partial write with no rollback corrupts state.

**M2 — N+1 / per-row DB work inside loops.**
The sweep flagged `db_*` calls inside `foreach` in several places. Most are **install/seed-time**
(`functions/install.php`, `func.php` language seeding) — acceptable, not hot paths. Worth
inspecting as request/cron hot paths: `novoton functions/hotels.php:529` (`foreach ($resorts ...)`),
`travel_core functions/hotels.php:204`, `travel_core functions/exchange_rates.php:183`. *Fix:* where
a loop issues one query per row, replace with a single `WHERE id IN (?n)` batch or a JOIN. (The
`CsCartFeatureAssignment` per-language loop and `SphinxProductFactory` per-language loop are bounded
by language count — acceptable.)

**M3 — CI coverage gaps.**
(a) **No `fgo_invoicing` / `smart-bill` PHPUnit jobs** in `ci.yml` — `fgo_invoicing` ships 9 test
files that **never run in CI** (and these addons are excluded from the `PHP Lint` `find` and
possibly the PHPStan paths). (b) The **integration suite is gated off** (`INTEGRATION_TESTDB_READY`)
— documented and by design, but it means no DB-backed test runs on PRs. *Fix:* add the two missing
PHPUnit jobs (cheap); confirm fgo/smart-bill are in `phpstan.neon` paths.

**M4 — Large procedural "god-files" beyond the documented tail.**
Besides the known deferred `novoton_price_compare.php` (1103) and `DiagnoseSearchCommand` (648),
these carry tangled responsibilities and no test/format coverage:
`sphinx func.php` (1223), `travel_core functions/hotels.php` (927), `novoton functions/hotels.php`
(822), `novoton_prices.php` controller (752), `add_to_cart.php` (708), `functions/install.php`
(707), `sphinx_holidays.php` controller (698), `novoton_tools.php` (669), `formatting.php` (650),
`email.php` (648). *Fix:* the same extract-collaborator-from-`src/` pattern already proven this
session applies — move pure logic out of these procedural files into tested `src/` services,
leaving the controller/function as a thin dispatcher.

### LOW

**L1 — `nofilter` volume + fragile tag re-introduction in `novoton_booking/search.tpl`.**
90 `nofilter` uses in novoton templates — verified mostly safe (escape-then-markup, or numeric
price HTML). But several do `…|escape:'html'|replace:'lt;pgt;':'<p>'|replace:'amp;':'&'|nl2br
nofilter` to selectively un-escape `<p>/<br>` from API-supplied hotel text. This is a **fragile,
hand-rolled allow-list** (string-replacing escaped fragments) that is easy to break and hard to
reason about. *Fix:* replace with a proper sanitizer (an allow-list HTML purifier or a Smarty
modifier that whitelists `<p>/<br>`) rather than escape-then-string-replace. **[verify]** the one
raw sink `{$booking_engine_html nofilter}` — confirm that HTML is server-built and never contains
unescaped user/API strings.

**L2 — Test-coverage ratio is thin in the shared core.**
`travel_core` has ~9 test files for ~72 `src/` classes; `sphinx` ~18 for 84. The recently-extracted
collaborators are tested, but many older core services/repositories are not. *Fix:* prioritize unit
tests for the highest-risk untested classes — anything doing money math, booking ownership,
security/validation, or API-response parsing. (A precise untested-class inventory is part of the
deferred deep-audit pass.)

**L3 — `escape:'html'` then `replace:'amp;':'&'`** (search.tpl) un-escapes ampersands globally,
which can resurrect `&#x...;`-style entities in attacker-influenced text. Low risk in a text node,
but tighten as part of L1's sanitizer fix.

## Deep pass — partial (novoton backend controllers, security)

> The 5-agent deep pass was throttled again by the account usage limit (resets **20:20 UTC**);
> 4 of 5 agents returned no findings. **One** deep sub-analysis completed: a full read of all
> **7 novoton `controllers/backend/` files**. Findings below — line numbers are the agent's and
> worth a quick confirmation. Still outstanding: sphinx + core controllers, all `Api/*Client`
> classes, and the correctness / performance / architecture / testing / frontend dimensions.

**Confirmed clean across the 7 novoton backend controllers** (`novoton_admin`, `novoton_alternatives`,
`novoton_bookings`, `novoton_diagnostic`, `novoton_holidays`, `novoton_hotels`, `novoton_seo_templates`):
no SQL injection (every request value reaches `db_*` via `?s/?i/?e/db_quote` placeholders; dynamic
`{$condition}`/`{$whereSql}` fragments only carry pre-bound values), **no path traversal** (the
`download_log` path is `basename()` + regex-filtered + `realpath()`-confined under `novoton_logs/`),
no open redirect (targets are static or pass `BookingAdminProvider::validateReturnUrl()` same-host
check), and no unescaped HTML/JS sink (the streaming HTML in `check_packages` is `htmlspecialchars`'d;
Smarty auto-escapes the rest).

**S1 — [Medium] CSRF on GET-triggered admin mutations.** `novoton_admin.php` `run_cron` (≈259-385)
and `update_prices` (≈23-82) fire on `$mode` regardless of HTTP method with no `fn_csrf_validate_request`.
`run_cron` is an explicit JSON/AJAX mutation endpoint and (per the agent) has **no `fn_check_permissions`
either** — only the Multivendor guard + an authenticated admin session — yet each allow-listed action
mutates the DB / triggers syncs. → CSRF-against-admin. *Fix:* require POST (CS-Cart then CSRF-protects),
or add an explicit token check; add a permission gate to `run_cron`.

**S2 — [Low–Medium] More GET-reachable DB mutations (permission-gated).** `novoton_hotels.php`
`sync_facilities` / `sync_hotel_facilities` / `save_facilities` / `check_packages` and
`novoton_holidays.php` `fix_tab` / `recompute_calendar_prices` mutate state on GET with
`fn_check_permissions('manage_catalog','update','admin')` but no CSRF token; `check_packages` is
literally a `method="get"` form doing DB upserts. *Fix:* make them POST-gated.

**S3 — [Low] Info disclosure.** `novoton_diagnostic.php` `health` (JSON) and `check` (text/plain)
dump infra/DB internals + exception messages over GET. Admin-only and non-HTML (no XSS), but verbose.
*Fix:* gate behind a debug flag / trim detail.

**S4 — [Info] Dead code.** `novoton_bookings.php` `_nvt_validate_return_url()` is defined but never
called (the real validation lives in `BookingAdminProvider`).

This sharpens the earlier **H2** into concrete endpoints; the consistent theme is **GET-reachable
admin state changes lacking CSRF tokens** — a hardening gap, not an open hole (all require an
authenticated admin).
