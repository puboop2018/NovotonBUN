================================================================================
  PRODUCT / HOTEL FEATURE AUDIT — NOVOTON vs SPHINX
  Which features can be shared by travel-core?
  Date: 2026-03-20
================================================================================


EXECUTIVE SUMMARY
-----------------

This audit inventories every product/hotel feature assigned by the Novoton and
Sphinx provider addons, compares them side-by-side, and determines which ones
can (and should) be managed by the shared travel-core addon.

Key findings:
  - Novoton defines 8 feature types; Sphinx uses only 4
  - 3 feature types are already shared via travel_core (stars, property_type,
    facility) — but only Sphinx actually uses the shared system
  - Novoton runs a parallel mapping table (hotel_feature_mappings) instead of
    using travel_core's travel_feature_map
  - 4 features could be added to travel_core: resort, board/meals, travel_group,
    beach_access
  - 2 canonical code sets exist in travel_core but are NOT assigned as product
    features by either provider: board and room_type

3. DETAILED DISCREPANCIES
================================================================================

3.1 Star Rating / Property Rating
----------------------------------
  Novoton:  Uses "property_rating" as feature_type in hotel_feature_mappings
            Codes: '1' through '5'
            Setting: addons.novoton_holidays.feature_id_property_rating

  Sphinx:   Uses "stars" as feature_type in travel_feature_map
            Codes: '1' through '5'
            Setting: addons.travel_core.feature_id_property_rating

  Issue:    Same feature, different names, different tables, different settings.
            Both resolve to the same CS-Cart feature_id in practice.

  Can share: YES — canonical codes are identical.


3.2 Board / Meals
------------------
  Novoton:  Assigns "meals" as M-type product feature (checkboxes)
            Codes: AI, UAI, FB, FB+, HB, HB+, BB, RO, SC (9 codes)
            Source: hotelData['boards'] array from Novoton API

  Sphinx:   Does NOT assign board/meals as a product feature.
            Board codes are only used in booking verification/search filtering.
            Normalizer has normalizeBoardCode() but SphinxFeatureAssigner
            does not call it.

  Travel Core: Has "board" canonical type with 7 codes (AI, UAI, FB, HB, BB, RO, SC)
               Missing FB+ and HB+ (Novoton-specific "plus" variants)

  Can share: YES — but requires:
    1. Add FB+ and HB+ to travel_core seed data
    2. Sphinx must start assigning board as product feature (if desired)
    3. Add feature_id_meals setting to travel_core (currently only in novoton)


3.3 Room Type
--------------
  Novoton:  Does NOT assign room_type as a product feature
  Sphinx:   Does NOT assign room_type as a product feature
  Travel Core: Has "room_type" with 8 canonical codes (SGL, DBL, TWIN, etc.)

  Status:   UNUSED as product feature by both providers. Canonical codes exist
            only for booking/search alias resolution.

  Can share: N/A — not used as product feature. May become relevant if
             room-level filtering is added to storefront search.


3.4 Facilities (Hotel + Room)
------------------------------
  Novoton:  Splits into 3 separate feature types:
            - hotel_facility (M) — hotel-level amenities
            - room_facility (M) — room-level amenities
            - beach_access (S) — beach proximity classification
            All use Novoton facility IDs as provider_code.
            Dynamic: unknown IDs auto-registered.

  Sphinx:   Single "facility" type (M) — all amenities in one bucket.
            Uses canonical codes from travel_feature_map.
            78 canonical codes mapped from 120+ Sphinx facility IDs.

  Travel Core: Has "facility" type with 102 canonical codes organized
               into 9 categories (food, wellness, parking, services,
               room amenities, outdoor, security, accessibility, smoking).

  Issue:    Novoton's 3-way split (hotel/room/beach) vs Sphinx's flat model.
            The canonical codes overlap but the CS-Cart features they map to
            may differ:
            - Sphinx: all facilities → one or a few CS-Cart features
            - Novoton: hotel_facility → feature X, room_facility → feature Y,
              beach_access → feature Z

  Can share: PARTIALLY — The canonical codes can be shared. The CS-Cart
             feature_id assignment per facility depends on the admin's
             preference (split vs flat). travel_feature_map already supports
             per-row cscart_feature_id, enabling both models.


3.5 Resort / Location
-----------------------
  Novoton:  "resort" feature type, S (SelectBox), DYNAMIC
            Codes: title-cased city names from novoton_resorts table
            Setting: addons.novoton_holidays.feature_id_resort

  Sphinx:   "resort" feature in SphinxFeatureAssigner
            Codes: title-cased destination_name from Sphinx API
            Setting: addons.travel_core.feature_id_location

  Travel Core: No "resort" canonical type defined in travel_feature_map.
               Has a setting (feature_id_location) but no seeded codes.

  Can share: YES — but resort is inherently dynamic (destination names are
             open-ended). Sharing requires:
             1. Add "resort" as a feature_type in travel_feature_map
             2. Seed common destination names
             3. Support dynamic registration of new resorts


3.6 Property Type
------------------
  Novoton:  "property_type" feature, S (SelectBox), STRICT
            Codes: hotel, motel, hostel, villa, apartment, boarding-house,
                   cabin, chalet, guest-house, resort (10 codes)

  Sphinx:   "property_type" feature, S (SelectBox)
            Codes: hotel, villa, apartment, resort, hostel, guest_house,
                   chalet, motel (8 codes)

  Travel Core: "property_type" canonical type with 8 codes:
               hotel, villa, apartment, resort, hostel, guest_house,
               chalet, motel

  Issue:    Novoton has 2 extra codes not in travel_core:
            - boarding-house (uses hyphen; travel_core has no equivalent)
            - cabin (not in travel_core)

  Can share: YES — add boarding_house and cabin to travel_core seed data.
             Normalize Novoton's "boarding-house" to "boarding_house" (no hyphen).


3.7 Travel Group
-----------------
  Novoton:  "travel_group" feature, M (Checkbox), STRICT
            Codes: "adults_only" (virtual) + facility IDs marked as
                   facility_type='travel_group' in novoton_facilities
            Used to flag adults-only hotels + travel group classification

  Sphinx:   NOT present. Sphinx API does not provide travel group data.

  Travel Core: NOT defined as a canonical type.

  Can share: YES (future) — if Sphinx or other providers start providing
             travel group data. Define "travel_group" in travel_feature_map
             with canonical codes: adults_only, family_friendly, seniors, etc.
             Currently Novoton-specific.


3.8 Beach Access
-----------------
  Novoton:  "beach_access" feature, S (SelectBox), STRICT
            Codes: facility IDs from novoton_facilities where
                   facility_type='beach_access'
            Represents proximity to beach (e.g., beachfront, 100m, 500m)

  Sphinx:   NOT present. Sphinx API does not provide beach access data
            as a distinct feature. Some beach info may exist in facilities.

  Travel Core: NOT defined as a canonical type.

  Can share: YES (future) — define "beach_access" canonical codes in
             travel_feature_map (beachfront, near_beach, no_beach, etc.)
             Currently Novoton-specific.

5. GAP ANALYSIS — What travel_core Needs
================================================================================

5.1 Missing Canonical Codes
-----------------------------
  Type            Missing Codes             Source
  --------------- ------------------------- -------------------------
  board           FB+, HB+                  Novoton-specific "plus"
                                            variants
  property_type   boarding_house, cabin      Novoton has these types
  resort          (entire type missing)      Dynamic — no seeded codes
  travel_group    (entire type missing)      adults_only + custom
  beach_access    (entire type missing)      beachfront, near_beach, etc.


5.2 Missing Feature Type Definitions
--------------------------------------
  travel_feature_map currently supports: board, room_type, stars,
  property_type, facility

  To add:
    - resort          (S) — dynamic destination names
    - travel_group    (M) — travel group classification
    - beach_access    (S) — beach proximity classification


5.3 Missing Addon Settings
----------------------------
  travel_core currently has settings for:
    feature_id_property_rating, feature_id_meals, feature_id_room_type,
    feature_id_property_type, feature_id_location

  Duplicated in novoton_holidays:
    feature_id_property_rating, feature_id_meals, feature_id_hotel_facility,
    feature_id_room_facility, feature_id_resort, feature_id_property_type,
    feature_id_travel_group, feature_id_beach_access

  To consolidate: Move all feature_id_* settings to travel_core.
  Add missing:
    - feature_id_hotel_facility  (or reuse facility with sub-types)
    - feature_id_room_facility   (or reuse facility with sub-types)
    - feature_id_travel_group
    - feature_id_beach_access


5.4 Missing Alias Registrations
---------------------------------
  Novoton currently registers aliases ONLY for board and room_type in
  travel_api_alias. It does NOT register aliases for:
    - stars (uses hotel_feature_mappings instead)
    - property_type (uses hotel_feature_mappings instead)
    - facilities (uses Novoton facility IDs in hotel_feature_mappings)
    - resort (uses hotel_feature_mappings instead)

  For full sharing, Novoton should register all its provider codes as
  aliases in travel_api_alias, just as Sphinx does.


5.5 Facility ID Mapping Gap
------------------------------
  Novoton:  Uses integer facility IDs (e.g., "42", "117") as provider_code
            in hotel_feature_mappings. These are Novoton-specific IDs.

  Sphinx:   Uses integer facility IDs that are mapped to canonical codes
            via travel_api_alias (e.g., Sphinx ID "31" → "fitness").

  Gap:      Novoton facility IDs are NOT mapped to travel_core canonical
            codes. To share the facility system, Novoton must register its
            facility IDs as aliases to canonical codes in travel_api_alias.

  Scope:    Novoton has ~200+ facility IDs. Each must be mapped to one of
            the 102 canonical facility codes (or new codes added as needed).

  1. Add FB+ and HB+ to board type in fn_travel_core_seed_feature_map()
  2. Add boarding_house and cabin to property_type
  3. Add "resort" feature_type to travel_feature_map (no seeded codes —

  4. Add "travel_group" feature_type with canonical codes:
     adults_only, family_friendly, seniors
  5. Add "beach_access" feature_type with canonical codes:
     beachfront, near_beach_100m, near_beach_500m, no_beach


Phase 2: Consolidate Addon Settings (MEDIUM effort)
-----------------------------------------------------
  1. Move feature_id_hotel_facility, feature_id_room_facility,
     feature_id_travel_group, feature_id_beach_access from
     novoton_holidays to travel_core addon settings
  2. Remove duplicate settings from novoton_holidays
  3. Update all references in novoton_holidays to read from
     travel_core settings instead


Phase 3: Register Novoton Aliases (MEDIUM effort)
---------------------------------------------------
  1. Map Novoton facility IDs to travel_core canonical facility codes.
     This is the biggest task — requires a mapping table of ~200 IDs.
  2. Register Novoton star rating codes (1-5) as aliases
  3. Register Novoton property type codes as aliases
  4. Register Novoton resort names as aliases (or use dynamic resolution)
  5. Add to fn_novoton_holidays_seed_travel_aliases()


Phase 4: Migrate Novoton to Shared Table (HIGH effort, FUTURE)
----------------------------------------------------------------
  1. Modify Novoton FeatureMapper to resolve canonical codes from
     travel_feature_map via FeatureMapper::resolve() instead of
     hotel_feature_mappings
  2. Keep hotel_feature_mappings for Novoton-specific metadata only:
     - strict/dynamic mode
     - variant_source (auto/manual)
     - mapping_source (seed/auto/manual)
     - last_synced_at
  3. Two-table approach: travel_feature_map for canonical resolution,
     hotel_feature_mappings for extended sync metadata
  4. Update the admin UI to show both layers


Phase 5: Enable Shared Features in Sphinx (OPTIONAL)
------------------------------------------------------
  1. Add board/meals assignment to SphinxFeatureAssigner
     (board codes already resolved by normalizer, just not assigned)
  2. Add travel_group detection if Sphinx API provides it
  3. Add beach_access classification if derivable from Sphinx facilities

================================================================================
8. CONCLUSION
================================================================================

The feature systems of Novoton and Sphinx overlap significantly but are
implemented through parallel mechanisms. The biggest wins come from:

  1. IMMEDIATE: Add missing canonical codes (FB+, HB+, boarding_house, cabin)
     to travel_core — eliminates code drift with almost zero effort.

  2. SHORT-TERM: Add resort, travel_group, and beach_access as feature types
     in travel_core — prepares the shared hub for full coverage.

  3. MEDIUM-TERM: Map Novoton's ~200 facility IDs to travel_core canonical
     codes — this is the largest effort but unifies the facility system across
     providers.

  4. LONG-TERM: Migrate Novoton to resolve from travel_feature_map while
     keeping hotel_feature_mappings for extended metadata — eliminates the
     dual-table problem.

The shared travel_core addon is well-positioned to become the single source
of truth for all product features. The canonical code set is already 80%
complete; the remaining 20% consists of Novoton-specific codes and two new
feature types that can be added incrementally.
