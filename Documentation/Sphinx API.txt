Sphinx API
API Documentation
This documentation will provide you with anything you need to know about the API. It covers all aspects from authentication to static data, from initiating searches to retrieving results and placing bookings.
Getting Started
To begin using our API and authenticate your requests, you will need the following:
    API endpoint: The base URL for accessing the API's resources. Our examples are created without such an endpoint. Make sure to properly add your own.
    API Key: Your unique key that you can use to access all the API features.

Url instanta dev:api.sphinx2.christiantour.dev.ploi.imementohub.com 
api key dev: 51|q3s6ZrK7212SFwQVBh5PkIOsPN9XS9WKJ7BtL9Puafaa6857

Using the API Key
To authenticate a request, simply include your API Key in the Authorization header with the Bearer scheme. Here's how to add the token to the request header using cURL:
Example request with Bearer Token
curl /api/v1/static/hotels \
  -H "Authorization: Bearer {token}"
Replace {token} with your API Key


Testing Authentication
The endpoints below have been created per your convenience, to easily test and confirm successful authentication.
You don't have to implement these endpoints in your application. They are only meant to help you validate your setup. Usually through a Postman Collection or something similar.

Ping
Feel free to use this endpoint to validate your setup. It will return a simple pong response.
This endpoint does not require authentication and its main purpose is to validate your domain configuration.
Receiving the expected pong response means that your domain is correctly configured and you can proceed to the next step.

"pong"

Rate Limiting
This API is meant to succesfully handle a very large number of legitimate requests. However, a couple of measures have been implemented to protect the API (and the resources behind it: servers, suppliers, etc.). This page describes the Rate Limiting behaviour and the actions you can take in the unlikely event you hit such limits.
Introduction

The API has been specifically designed to be as fast and as reliable as possible. It is specifically built to handle very large applications and websites, with hundreds of parallel requests. On the other hand, there are some endpoints that are more sensitive to high loads and, as such, they required some additional protection.
While it is very unlikely that you will ever hit the rate limits, it is important to understand how they work, what you can do to avoid them and what to do if you hit them.
Rate Limiting Basics

In the most simple scenario, there are predefined rate limits on some of the endpoints. One such example is the ping endpoint, which has been designed with a very low rate limit so that it can be used to test rate limit scenarios. The rate limit for this endpoint is 10 requests per minute.
Endpoint: /api/v1/ping
Rate Limit: 10 requests per minute
The Rate Limiting strategies that are implemented for the other endpoints are much more complex. They are based on the following principles:
    Support a very large number of legitimate requests
    Protect the API from usage that is not in line with this documentation

For example, the verify endpoints have Rate Limits that, among other things, also consider the search-id. This allows for a very large number of requests for different searches, while still protecting the API from extensive parallel requests for the same search. The limits are way above any legitimate usage by actual human behaviour.
Rate Limit Responses

    HTTP 200 OK: The request was successful and no further action is required. The response body will contain the requested data. You can optionally inspect the X-RateLimit-Remaining header to determine the number of requests remaining within the current time frame.
    HTTP 429 Too Many Requests: The rate limit has been exceeded. The response body will not contain any data. Retry after the X-RateLimit-Reset timestamp or after Retry-After seconds.

Rate Limit Headers
When you receive a HTTP 429 Too Many Requests response, you can check the X-RateLimit-Reset header to determine when to retry the request. Here are some other useful headers that you can use to monitor your rate limit usage:
    X-RateLimit-Limit: The maximum number of requests allowed per time frame.
    X-RateLimit-Remaining: The number of requests remaining within the current time frame.
    X-RateLimit-Reset: The timestamp when the rate limit will reset.
    Retry-After: The time, in seconds, until the rate limit will reset.
Best Practices
    Always check the status code on the Response and act accordingly.
    If you receive a HTTP 429 Too Many Requests response, check the X-RateLimit-Reset header to determine when to retry the request.
    Avoid continuing to send requests after receiving a HTTP 429 Too Many Requests response. This will only lead to bad user experience.

Testing
Feel free to use the Ping endpoint to test your application against the Rate Limiting behaviour. There is a hard limit of 10 requests per minute on this endpoint.

Caching Data

Caching static data is encouraged and highly recommended! The API exposes dedicated endpoints for static data. However, there are endpoints that have not been designed for caching purposes. Please read below about the best practices of data caching.
Caching Search Results
Please use the API responsibly!
All our search endpoints retrieve live data from our suppliers. As such, please avoid using the live search endpoints to build cache databases.

While we fully understand the need to provide fast responses to your users, we must also ensure that our suppliers are not overloaded with requests. If building a cached results database is mandatory for your application, please reach out and we are more than open to find a solution that accommodates your needs.
Caching Static Data
We fully encourage the cache of static data, such as Hotels, Circuits, Destinations or Images.
Feel free to schedule regular calls to the static data endpoints and cache the results for your application. For the bests performance you can schedule your cronjobs during low load periods your servers (usually during night time).
Because some datasets can be rather large and most data does not change very often, you can also use the updated_since parameter to only retrieve the data that was changed since the given date.
Caching Images
Travel content is very visual. As such, images are a very important part of the data we provide and we constantly strive to provide top quality images for our resources (eg: hotels).
By being granted access to this API, you have also been granted access to the original images, without any watermarks from the supplier. Read below to find out how to retrieve the original images.

Ignoring Domains
Travel agencies heavily rely on each other and share their products and offers to efficiently expand their portfolios. Operating in such an interconnected environment leads to serious challenges when it comes to optimizing search queries and response times.
It is not uncommon to see suppliers receiving duplicate requests or even the appearance of infinite loops between various suppliers.
We introduce the ignore_domains parameter to address such scenarios and to successfully protect and safeguard all of our partners. We strongly encourage you to use this parameter and join a professional elite not only focused on delivering qucick and fast search results, but also in protecting the technical ecosystem.
Duplicated Request
Let's imagine a simple scenario where an OTA (Agency X) has two main suppliers (A and B). Supplier A also relies on Suppliers B and C to expand its portfolio.
Without any other intervention, this very common arrangement will easily lead to duplicated requests on Supplier B, leading to the following outcomes:
•	Higher, unnecessary load on Supplier B's servers.
•	Increased wait times on Agency X to retrieve all offers.
•	The need for a deduplication process on Agency X, to avoid display of duplicate results.
•	Redundant payloads over the network and redundant, futile processing, as some of the results will be discarded.
 
The example above is a very simple one, yet very common. In live environments the situation is much more complex and the duplication might occur at deeper levels, being much harder to detect and avoid. In extreme scenarios one can easily imagine how infinite loops could occur, leading to a significant overloads and even system crashes.
Unique Request
By using the ignore_domains parameter, the situation above can be easily avoided. Sphinx will gracefully stop propagating requests to the suppliers that have already been interrogated, leading to a much more efficient and optimized search process.
Additionally, Sphinx will forward the ignore_domains parameter to all of its suppliers that support this feature, so that they can also avoid unnecessary processing and further optimize the entire process.

 
This approach has numerous advantages as outlined above. The most important thing to note is that through this approach, each supplier has direct access to its main contracts, skipping any unnecessary intermediaries.

Other alternatives
We are always open to implement other alternatives to address this issue. If you have any suggestions, please contact us and we will be more than happy to discuss them with you!

Debugging
Everyone is happy when things go well! We are here to support when the unexpected happens.

Inspect Your Responses
We strive to provide helpful error messages and responses. However, sometimes these might not be enough to understand what went wrong and what are the proper ways to fix it.
In these cases, please reach out to us and we will be happy to investigate together. To help us get started, please mention the X-Request-ID header from the response.
API Changes
The travel industry is an ever evolving environment. We aim to provide a very stable API while, in the mean time, be flexible enough to accomodate rapid adjustments to market and business requirements and expectations.
Please find below how we are constantly updating our API without introducing breaking changes so that your app will always be stable and performant.
We adopt a strong No Breaking Changes policy.
Feature Updates
All updates marked as a Feature Update are designed to be fully Backwards compatible.
This means that even though we are adding new features to the API, your existing code will continue to work as expected. You can safely update your application to the latest version of the API at a later point and without any risk of breaking your existing code.
If you consider that any of the changes listed below could be considered a breaking change for your implementation, please contact us.
The following changes are considered Feature Updates:
•	New Properties: Adding new properties to existing objects is considered a backwards compatible change. This can apply to Requests, Responses, or any other object inside them.
•	New Optional Parameters: Any new optional parameters will not change the initial behavior of the API.
•	New Endpoints: Adding new endpoints is considered a backwards compatible change as long as they are not required by a complete booking flow.
Breaking Changes
We have designed our API to be as stable as possible while still allowing us the much needed flexibility.
We take extra care in avoiding any breaking changes. We list below all the changes that we consider to lead to breaking updates. If you consider we should add any other changes to this list, please contact us.
However, if we ever plan to introduce breaking changes, we will communicate them well in advance and provide a migration path to help you update your application. Please register as a developer to receive updates on any potential breaking changes.
The following updates are considered Breaking Changes:
•	New Required Parameters: Adding new required parameters to an existing endpoint is considered a breaking change. This is because the API will not be able to process the request without the new required parameters.
•	Changing Properties: Any of the following changes on a property already documented is considered a breaking change: 
o	Changing the name of a property
o	Removing an existing property
o	Changing the type of a property value
o	Changing the format of a property value
•	Removing an Endpoint: Removing a previously documented endpoint is considered a breaking change, even if that endpoint was not mandatory for a complete booking flow.
Consider registering as a developer to receive updates about future updates.
Destinations
Destinations are one of the most important resources in our API. They are used to initiate searches and to filter results. Destinations are organized in a tree structure, where each destination can have a parent destination. The continents are at the top level.
Recommended flow
We highly encourage regularly retrieving destinations and storing them locally, in your database. Feel free to setup a cronjob that runs every night. We encourage using the updated_since parameter to only retrieve destinations that have been updated since the last time you retrieved them. Using this parameter will greatly reduce the amount of data that needs to be transferred and, as such, you can run the cronjob more often (eg: every hour).
Having an up to date list of destinations will allow you to initiate searches and explore all the offers that the API has to offer.
Destination mapping
We understand that for a robust application, that integrates more suppliers, mapping destinations is a must. As such, we are doing our best to provide you with all the information you need to successfully map our destinations to your own destinations.
Below you can find a couple of details that could help in improving and easing the mapping process.
For most of our destinations you will receive a geoname_id. This is the id of the destination in the GeoNames database. Chances are that you already use a centralized destinations database. We found the geoname_id to be a very reliable and robust method to accurately map the destinations.
We also provide the latitude and longitude for most destinations. This mapping approach is more suitable if your application relies heavily on geolocated destinations.
The international country_code and a parent_id, belonging to a strict tree structure will always be provided. Once a parent destination has been mapped (eg: a continent or a country), all the child destinations should be easier to reliably map to your own destinations.
The continents will have a country_code made of 3 letters, starting with X. (eg: XAF for Africa).
We are always open to any suggestions on how to improve the mapping process. Feel free to contact us using the Contact page if you would like us to also include other details to better describe the destinations.

Hotels

Hotels are another central element of the API. Read below on how to easily retrieve them and store them locally on your system.
Recommended flow

As with destinations, we highly encourage regularly retrieving hotels and storing them locally, in your database. Feel free to setup a cronjob that runs every night. We encourage using the updated_since parameter to only retrieve hotels that have been updated since the last time you retrieved them. Using this parameter will greatly reduce the amount of data that needs to be transferred and, as such, you can run the cronjob more often (eg: every hour).
Hotel mapping

While hotel mapping is optional, as it does not necessarily impact the API integration, in our experience mapping hotels leads to a much better outcome. With this approach search results could be easily aggregated and hotel duplication in search results can be avoided.

In order to help with hotel mapping, we strive to provide accurate geolocation for each hotel, whenever this is possible.

Additionally, when available we will provide with external, independent identifiers for each hotel. (eg: Giata id)

Experiences

Experiences are another central element of the API. Read below on how to easily retrieve them and store them locally on your system.
Recommended flow

As with destinations, we highly encourage regularly retrieving experiences and storing them locally, in your database. Feel free to setup a cronjob that runs every night. We encourage using the updated_since parameter to only retrieve experiences that have been updated since the last time you retrieved them. Using this parameter will greatly reduce the amount of data that needs to be transferred and, as such, you can run the cronjob more often (eg: every hour).

POST/api/v1/hotels/book
Book offer

Use this endpoint to book an offer.
Request attributes

    Name
        offer_id
    Type
        string
    Description

        The corresponding offer_id returned by the Get Results call.

    Name
        reference_code
    Type
        string (optional)
    Description

        A unique identifier internal identifier from your system. If sent, it will allow you to retrieve a booking by its reference id.

    Name
        price
    Type
        float
    Description

        The price of the offer returned by the Get Results call or the price returned by the Verify Offer call.

        The value of this field should match the selling_price from the pricing object in the response.

        This field is validated against the price in our system. If the price is not the same, the booking will fail.

    Name
        currency
    Type
        string
    Description

        The currency of the offer returned by the Get Results call.

        This field is validated against the currency in our system. If the currency is not the same, the booking will fail.

    Name
        occupancy
    Type
        array of Occupancy objects
    Description

        An array describing the distribution of guests in the rooms.

Occupancy

The distribution of guests in the rooms must be the same as the one originally requested in the Search request and returned by the Get Results call.

    Name
        room_code
    Type
        string
    Description

        The room code of the offer returned by the Get Results call. The room code might be similar between rooms.

    Name
        guests
    Type
        array of Guest objects
    Description

        The guests that will occupy the room.

        For each guest, the following fields are required:
            first_name
            last_name
            birth_date
            gender

        The birth_date will validate the age of the guest at checkout and it has to correspond to the age requested in the Search request. If the age is not the same, the booking will fail, as this might lead to a different price.

        The birth_date is mandatory and must be accurate for children, as it has to be certified by the hotel at check-in.

        For compliance reasons, the birth_date is also required for adults. Still, if these details are not available, a common practice is to send a default date, for a 30 years old person at the time of check-in.

        The gender field is mandatory and must be one of the following values: m or f.

    Name
        order_id
    Type
        int
    Description

        The unique identifier of the booking from our system.

    Name
        contract_id
    Type
        int|null
    Description

        The unique identifier of the contract from our system.

        This property can be null if the contract has not yet been generated. contract_id can also be obtained through the Orders endpoint

    Name
        booking_confirmation_number
    Type
        string
    Description

        The confirmation number of the booking provided by the supplier.

        Most of the time it is the code that appears on the travel voucher.

    Name
        reference_code
    Type
        string|null
    Description

        The unique internal identifier from your system.

        You will receive this field only if you sent a reference_code in the request.

    Name
        status
    Type
        enum
    Description

        The status of the booking. All successful bookings will have the status confirmed. Can be one of the following values:
            confirmed - The booking is confirmed.
            unconfirmed - The booking is not confirmed.
            on_request - The booking is on request, need to be confirmed by the supplier.
            processing - The booking is under processing. You should wait for a few seconds and then check the status again using the Orders endpoint. When status is not processing, you can consider the booking is completed. If the status is still processing, you should check again after 20 seconds.
            canceled - The booking is canceled.
            errored - The booking was not confirmed.

        If the booking fails, you will receive an error message with the reason.

Pricing

This object contains all the pricing details for a given offer.

    Name
        marketing_price
    Type
        float
    Description

        This is the initial price of the offer, before any discounts.
        (Example: 5000.00)

        It is usually the price that is displayed as cut-through to the end user.

    Name
        discount
    Type
        float
    Description

        This is price reduction that is applied from the marketing_price.
        (Example: 1000.00)

    Name
        selling_price
    Type
        float
    Description

        This is the actual recommended selling price of the offer.
        (Example: 4000.00)

        In other terms, this is the price that the end user is expected to pay.

        selling_price = marketing_price - discount

    Name
        commission
    Type
        float
    Description

        This is amount that the reseller is entitled to.
        (Example: 400.00)

    Name
        supplier_price
    Type
        float
    Description

        This is price that needs to be paid by the reseller.
        (Example: 3600.00)

        supplier_price = selling_price - commission

    Name
        currency
    Type
        string
    Description

        All values above are expressed in the currency specified by this attribute.
        (Example: RON)

        This is, most often, the currency in which the search has been initiated.

    Name
        taxes
    Type
        array of objects
    Description

        These taxes are already included in the selling_price value.

        See: Taxes

    Name
        additional_fees
    Type
        array of objects
    Description

        These fees are not included in the selling_price value. The end user is expected to pay them separately during travel or at the hotel.

        See: Additional Fees

Overview

This page describes the Search process and how to retrieve asynchronous results from the API.

The API can respond within seconds with results to a search request, even if it may take a couple of tens of seconds to provide all offers. This is achieved by using a cursor-based pagination system. The cursor is a token that represents the first page of results. You can use this token to retrieve the next page of results, and so on, until the end of the search results is reached.

Read below to learn more about the search flow and how to retrieve results asynchronously.
Search Flow

After placing a Search request you will immediately receive a cursor token.

Use that cursor to place a Get Results request and receive the first page of results. Each results page will contain a new cursor token that you can use to retrieve the next page of results. Repeat this process until you receive a null cursor token, which indicates that you have reached the end of the search results.
Timing

Feel free to place a Get Results request as soon as you receive the next cursor token. The API will respond immediately if there are results available. When there are no results available yet, the API might wait a couple of seconds for new results to become available before responding.
Number of results

The number of results on a page can vary greatly, depending on the number of offers available at that particular moment. There are frequent scenarios where pages might contain no results at all. This is normal and expected behavior. It means that we are still waiting for replies from some of the suppliers.

You are expected to continue polling even if you receive pages with no results. When there are no more results to be processed the API will respond with cursor: null. That is the cue that you have reached the end of the search results.
User experience

This entire flow has been designed to provide the best possible user experience. You are expected to display the partial results to the user as soon as they become available, without having to wait for the entire search process to finish.

If there is any impediment to do so, please let us know and we will do our best to find a solution that works best for you.
Booking flow

Some Offers may require verification before they can be booked. This is indicated by the must_verify flag. If this flag is set to true, you will need to Verify the offer before you can Book it.

The verification process is an expensive operation that can take up to 10 seconds to complete. You are expected to perform this operation only when the user is ready to book the offer. Various Rate Limitting mechanisms are in place to prevent abuse of the Verify endpoint. Please use it only for legitimate requests.

POST/api/v1/packages/book
Book offer

Use this endpoint to book an offer.
Request attributes

    Name
        offer_id
    Type
        string
    Description

        The corresponding offer_id returned by the Get Results call.

    Name
        reference_code
    Type
        string (optional)
    Description

        A unique identifier internal identifier from your system. If sent, it will allow you to retrieve a booking by its reference id.

    Name
        price
    Type
        float
    Description

        The price of the offer returned by the Get Results call or the price returned by the Verify Offer call.

        The value of this field should match the selling_price from the pricing object in the response.

        This field is validated against the price in our system. If the price is not the same, the booking will fail.

    Name
        currency
    Type
        string
    Description

        The currency of the offer returned by the Get Results call.

        This field is validated against the currency in our system. If the currency is not the same, the booking will fail.

    Name
        occupancy
    Type
        array of Occupancy objects
    Description

        An array describing the distribution of guests in the rooms.

    Name
        additional_services
    Type
        array of string
    Description

        An array of additional services that you added to the offer using the Customize Offer call.

        You should not include the services that are already included in the offer (with the selected flag set to true and the type set to mandatory).

Occupancy

The distribution of guests in the rooms must be the same as the one originally requested in the Search request and returned by the Get Results call.

    Name
        room_code
    Type
        string
    Description

        The room code of the offer returned by the Get Results call. The room code might be similar between rooms.

    Name
        guests
    Type
        array of Guest objects
    Description

        The guests that will occupy the room.

        For each guest, the following fields are required:
            first_name
            last_name
            birth_date
            gender

        The birth_date will validate the age of the guest at checkout and it has to correspond to the age requested in the Search request. If the age is not the same, the booking will fail, as this might lead to a different price.

        The birth_date is mandatory and must be accurate for children, as it has to be certified by the hotel at check-in.

        For compliance reasons, the birth_date is also required for adults. Still, if these details are not available, a common practice is to send a default date, for a 30 years old person at the time of check-in.

        The gender field is mandatory and must be one of the following values: m or f.

    Name
        order_id
    Type
        int
    Description

        The unique identifier of the booking from our system.

    Name
        contract_id
    Type
        int|null
    Description

        The unique identifier of the contract from our system.

        This property can be null if the contract has not yet been generated. contract_id can also be obtained through the Orders endpoint

    Name
        booking_confirmation_number
    Type
        string
    Description

        The confirmation number of the booking provided by the supplier.

        Most of the time it is the code that appears on the travel voucher.

    Name
        reference_code
    Type
        string|null
    Description

        The unique internal identifier from your system.

        You will receive this field only if you sent a reference_code in the request.

    Name
        status
    Type
        enum
    Description

        The status of the booking. All successful bookings will have the status confirmed. Can be one of the following values:
            confirmed - The booking is confirmed.
            unconfirmed - The booking is not confirmed.
            on_request - The booking is on request, need to be confirmed by the supplier.
            processing - The booking is under processing. You should wait for a few seconds and then check the status again using the Orders endpoint. When status is not processing, you can consider the booking is completed. If the status is still processing, you should check again after 20 seconds.
            canceled - The booking is canceled.
            errored - The booking was not confirmed.

        If the booking fails, you will receive an error message with the reason.

Pricing

This object contains all the pricing details for a given offer.

    Name
        marketing_price
    Type
        float
    Description

        This is the initial price of the offer, before any discounts.
        (Example: 5000.00)

        It is usually the price that is displayed as cut-through to the end user.

    Name
        discount
    Type
        float
    Description

        This is price reduction that is applied from the marketing_price.
        (Example: 1000.00)

    Name
        selling_price
    Type
        float
    Description

        This is the actual recommended selling price of the offer.
        (Example: 4000.00)

        In other terms, this is the price that the end user is expected to pay.

        selling_price = marketing_price - discount

    Name
        commission
    Type
        float
    Description

        This is amount that the reseller is entitled to.
        (Example: 400.00)

    Name
        supplier_price
    Type
        float
    Description

        This is price that needs to be paid by the reseller.
        (Example: 3600.00)

        supplier_price = selling_price - commission

    Name
        currency
    Type
        string
    Description

        All values above are expressed in the currency specified by this attribute.
        (Example: RON)

        This is, most often, the currency in which the search has been initiated.

    Name
        taxes
    Type
        array of objects
    Description

        These taxes are already included in the selling_price value.

        See: Taxes

    Name
        additional_fees
    Type
        array of objects
    Description

        These fees are not included in the selling_price value. The end user is expected to pay them separately during travel or at the hotel.

        See: Additional Fees

Payment Terms

The Payment Terms describe the contractual obligations regarding payments for the given offer.

Failure to comply with these terms might result in the automatic cancellation of the booking.

The value of each rule is expressed in the currency of the offer and it represents the total amount that needs to be paid up until the specified date.

    Name
        is_loaded
    Type
        boolean
    Description

        Some suppliers might not provide the payment terms for the offers during the initial search request. In order to distinguish between scenarios where there are no actual terms for a given offer and scenarios where the terms are not yet available, we have introduced this flag.

        After calling the verify endpoint or the customize endpoint, the payment terms will be updated accordingly.

    Name
        text
    Type
        string
    Description

        If, for any reason, we are unable to properly parse the payment terms, we will provide the raw text in this attribute.

    Name
        rules
    Type
        array of objects
    Description

        The list of rules that need to be followed in order to comply with the payment terms. They are sent in chronological order.

Rule

    Name
        until
    Type
        date (YYYY-MM-DD)
    Description

        The date until which the payment needs to be made.

    Name
        value
    Type
        float
    Description

        The total amount that needs to be paid up until the specified date.

        A unique identifier internal identifier from your system. If sent, it will allow you to retrieve a booking by its reference id.

    Name
        price
    Type
        float
    Description

        The price of the offer returned by the Get Results call or the price returned by the Customize Offer call.

        The value of this field should match the selling_price from the pricing object in the response.

        This field is validated against the price in our system. If the price is not the same, the booking will fail.

    Name
        currency
    Type
        string
    Description

        The currency of the offer returned by the Get Quote call.

        This field is validated against the currency in our system. If the currency is not the same, the booking will fail.

    Name
        occupancy
    Type
        array of Occupancy objects
    Description

        An array describing the distribution of guests in the rooms.

Occupancy

The distribution of guests in the rooms must be the same as the one originally requested in the Get Quote request and returned by the response.

    Name
        room_code
    Type
        string
    Description

        The room code of the offer returned by the Get Results call. The room code might be similar between rooms.

    Name
        guests
    Type
        array of Guest objects
    Description

        The guests that will occupy the room.

        For each guest, the following fields are required:
            first_name
            last_name
            birth_date
            gender

        The birth_date will validate the age of the guest at checkout and it has to correspond to the age requested in the Get Quote requestt. If the age is not the same, the booking will fail, as this might lead to a different price.

        The birth_date is mandatory and must be accurate for children, as it has to be certified by the hotel at check-in.

        For compliance reasons, the birth_date is also required for adults. Still, if these details are not available, a common practice is to send a default date, for a 30 years old person at the time of check-in.

        The gender field is mandatory and must be one of the following values: m or f.

Payment Terms

The Payment Terms describe the contractual obligations regarding payments for the given offer.

Failure to comply with these terms might result in the automatic cancellation of the booking.

The value of each rule is expressed in the currency of the offer and it represents the total amount that needs to be paid up until the specified date.

    Name

    Type
        boolean
    Description

        After calling the quote endpoint, the payment terms will be updated accordingly. This flag indicates if the payment terms are available.

    Name

        array of objects
    Description

Book offer

Use this endpoint to book an experience offer. You must provide the offer_id returned by the Get Quote call
Request attributes

    Name

    Type
        string
    Description

        The corresponding offer_id returned by the Get Results call.

    Name
        reference_code
    Type
        string (optional)
    Description

        A unique identifier internal identifier from your system. If sent, it will allow you to retrieve a booking by its reference id.

    Name
        price
    Type
        float
    Description

        The price of the offer returned by the Get Results call or the price returned by the Customize Offer call.

        The value of this field should match the selling_price from the pricing object in the response.

        This field is validated against the price in our system. If the price is not the same, the booking will fail.

    Name
        currency
    Type
        string
    Description

        The currency of the offer returned by the Get Quote call.

        This field is validated against the currency in our system. If the currency is not the same, the booking will fail.

    Name
        occupancy
    Type
        array of Occupancy objects
    Description

        An array describing the distribution of participants.

Occupancy

The distribution of participants must be the same as the one originally requested in the Get Quote request and returned by the response.

For each participant, the following fields are required:

    first_name
    last_name
    birth_date
    gender

The birth_date will validate the age of the guest at checkout and it has to correspond to the age requested in the Get Quote requestt. If the age is not the same, the booking will fail, as this might lead to a different price.

The birth_date is mandatory and must be accurate for children, as it has to be certified by the hotel at check-in.

For compliance reasons, the birth_date is also required for adults. Still, if these details are not available, a common practice is to send a default date, for a 30 years old person at the time of check-in.

The gender field is mandatory and must be one of the following values: m or f.
