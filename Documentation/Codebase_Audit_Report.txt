================================================================================
  2. CRITICAL ISSUES (FIXED)
================================================================================

2.1 BUG: $mode overwrite in travel_bookings.php
-------------------------------------------------
  File:   addon-travel-core/.../controllers/backend/travel_bookings.php
  Line:   23 (was: $mode = $_REQUEST['mode'] ?? 'manage';)
  Impact: CS-Cart auto-sets $mode from dispatch URL. Overwriting it causes
          404 errors — same bug pattern fixed in sphinx_holidays earlier.
  Fix:    Removed the $mode overwrite line. Added restricted admin guard.

2.2 BUG: $mode overwrite in travel_booking.php (frontend)
----------------------------------------------------------
  File:   addon-travel-core/.../controllers/frontend/travel_booking.php
  Line:   24 (was: $mode = $_REQUEST['mode'] ?? '';)
  Impact: Same $mode overwrite pattern breaking CS-Cart dispatch routing.
  Fix:    Removed the $mode overwrite line.

2.3 BUG: $mode overwrite in novoton_bookings.php
--------------------------------------------------
  File:   addon-novoton-holidays/.../controllers/backend/novoton_bookings.php
  Lines:  18-19 (was: if (empty($mode)) { $mode = 'manage'; })
  Impact: Overwrites CS-Cart's auto-set $mode when dispatch URL has no mode.
  Fix:    Removed the $mode overwrite. Added restricted admin guard.

2.4 SECURITY: Hardcoded API credentials in addon.xml
------------------------------------------------------
  File:   addon-novoton-holidays/.../addon.xml
  Lines:  63-83
  Values: api_url=b2b.allinclusivebg.com, api_key=TEST-TEST-TEST-TEST-TEST,
          api_id=713, api_user=EHROM117, api_password=EUP359YJX (plaintext!)
  Impact: Credentials visible in source code / version control.
  Fix:    Removed all default values. Admin must configure after install.

2.5 SECURITY: Missing input validation in travel_feature_mappings.php
----------------------------------------------------------------------
  File:   addon-travel-core/.../controllers/backend/travel_feature_mappings.php
  Lines:  152-154
  Issue:  match_type accepted any string, api_source not sanitized.
  Fix:    Added whitelist validation for match_type ['exact','prefix','contains'].
          Sanitized api_source with preg_replace to allow only [a-z0-9_].
          Fixed AJAX exit pattern to use fn_echo() instead of raw exit.

================================================================================
  4. MEDIUM SEVERITY (ASSESSED / DEFERRED)
================================================================================

4.1 PriceInfoCalculator.php — 1,451 lines  [DEFERRED]
  File:   addon-novoton-holidays/.../PriceInfoCalculation/PriceInfoCalculator.php
  Status: Complex but working correctly. Splitting would be a refactoring
          project with regression risk. No functional issue.
  Action: Document for future refactoring. Consider splitting into
          BasePriceCalc, FeeApplier, DiscountApplier, CommissionCalculator.

4.2 Language hardcoding in FeatureMapper  [ACCEPTED]
  File:   addon-novoton-holidays/.../Services/FeatureMapper.php:121,125-127
  Status: Uses display_name_ro / display_name_en columns from DB schema.
          The getActiveLanguages() method (line 574) already queries DB
          dynamically for variant creation. The column-level "hardcoding"
          is a schema limitation, not a code issue.
  Action: Would need schema migration to support additional languages.
          Not a bug — accepted as-is.

4.3 Duplicate feature variant functions  [ACCEPTED]
  Files:  novoton func.php (8 functions), travel_core func.php (5 functions)
  Status: CS-Cart convention requires function names matching pattern
          fn_settings_variants_addons_{addon}_{setting_id}().
          Cannot be auto-generated due to CS-Cart's auto-discovery by name.
  Action: Accepted as necessary boilerplate.

4.4 API error handling inconsistency  [DEFERRED]
  Status: Some Novoton API clients return false, others throw exceptions.
          Not causing issues in practice — callers handle both.
  Action: Standardize in a future API client refactoring.

4.5 BookingRepository inline GuestDataNormalizer  [ACCEPTED]
  File:   addon-novoton-holidays/.../Repository/BookingRepository.php:36
  Status: Constructor accepts ?GuestDataNormalizer with fallback to new().
          This is a valid nullable DI pattern, not a bug.
  Action: No change needed.
