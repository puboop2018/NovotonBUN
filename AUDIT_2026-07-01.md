## Executive summary
The codebase is **mature and well-hardened**: PHPStan L10 on an **empty** baseline (self-enforcing), god-classes decomposed into tested collaborators, TLS on with timeouts, API creds masked in logs, no hardcoded secrets, booking endpoints ownership-scoped (no IDOR), cron access-key gated, SQL parameterized. This audit is mostly *hardening* — with **one architecturally-significant correctness defect** worth fixing first.

**Fix first:** the Novoton booking-submission "transaction" provides no atomicity (C1). Everything else is High/Medium hardening or scale work.

## Findings

### C1 — [High→Critical · VERIFIED] Booking-submission transaction is a no-op; rollback never fires
`novoton_holidays/src/Services/BookingSubmissionService.php:128-208` + `src/Repository/BookingRepository.php:217/242/312`
`submitOrder()` opens `START TRANSACTION` (128) then, per room-group, calls `recordBuilder->persist()` → `BookingRepository::create()`/`update()`, **each of which opens its own `START TRANSACTION`/`COMMIT`** (217/225, 242/251). MySQL has no nested transactions: the inner `START` **implicitly commits** the outer one, and the inner `COMMIT` closes it. So when a later group's API call throws, the `ROLLBACK` at 198/218/235 runs with **no active transaction and undoes nothing** — earlier groups' bookings stay committed. The intended all-or-nothing multi-room submission does not exist; the rollback (and its explanatory comment at :200) is dead code.
*Failure scenario:* 2-room order, group 1 books + reserves, group 2's Novoton call fails → DB keeps a confirmed group-1 booking + `travel_bookings` mirror alongside a failed group-2 record, with no atomic unwind. Financial exposure if payment is captured for the whole order.
*Fix:* Repositories must not open their own transaction when callable inside one — remove the inner `START/COMMIT` and let the caller own it, or adopt Sphinx's `withTransaction()` nesting-aware helper. Then restructure `submitOrder` to **persist+COMMIT the booking, call the API OUTSIDE any transaction, then record the response in a separate short transaction** (this also prevents holding row locks across the network call once nesting is fixed). Unblocks C1-related test gaps (H5).

### H1 — [High · VERIFIED] "Batch" upserts are row-by-row N+1 (sync cron)
`sphinx_holidays/src/Repository/HotelRepository.php:62-157` and `DestinationRepository.php:43-77`
Despite the name, `upsertBatch()` loops one `INSERT … ON DUPLICATE KEY UPDATE` per row (confirmed: `foreach` at :62, query at :68). 100k hotels → 100k round-trips; the stream-and-chunk design upstream is defeated.
*Fix:* true multi-row `INSERT … VALUES (…),(…) ON DUPLICATE KEY UPDATE col=VALUES(col)` (the `IF(...)` update clauses work with multi-row VALUES).

### H2 — [High · Agent-reported, patterns spot-confirmed] Missing indexes / non-sargable predicates
- `sphinx HotelRepository.php:447` `markInactiveBefore` filters `(country_code, sync_status, last_synced_at)` with no matching composite index → range-scans + row-locks every active hotel in a country each sync. **Add `(country_code, sync_status, last_synced_at)`.**
- `sphinx SphinxProductFactory.php:154-198` dedup on `name` (unindexed) and `ROUND(latitude,3)/ROUND(longitude,3)` (non-sargable) → full scans per new hotel. **Index `name`; store rounded-coord columns.**
- `sphinx DestinationRepository.php:255-263/510-521` `LOWER(name)`/`LOWER(full_path) LIKE` → full scans of up to ~200k rows on request paths. **Normalized lowercase column or CI collation.**

### H3 — [High · VERIFIED] `ON DELETE RESTRICT` FK makes booked hotels undeletable
`novoton_holidays/functions/install.php:578-593` (`fk_nb_hotel_id`, `fk_nar_hotel_id`)
Bookings/alt-requests reference `novoton_hotels` with RESTRICT, so `HotelRepository::delete()`'s final `DELETE FROM novoton_hotels` throws whenever history exists → the whole delete transaction rolls back and the hotel can never be removed or re-synced (delete-then-reinsert reconciliation breaks). This is the same constraint behind the earlier bulk-delete workaround.
*Fix:* bookings are historical and must outlive the master row — change to `ON DELETE SET NULL` (nullable `hotel_id`) or drop the FK and keep `hotel_id` as a denormalized snapshot (the booking already stores name/dates/prices).

### H4 — [High · VERIFIED (with nuance)] Provider FK asymmetry / inconsistent integrity model
`novoton` installs 5 FKs (`fn_novoton_holidays_setup_db`); **`sphinx` installs none** and instead relies on a `CleanupCommand` cron to unlink orphaned `product_id` refs. Two different integrity philosophies across sibling providers; sphinx's destination→hotel and whitelist relationships have neither FK nor guaranteed cleanup.
*Fix:* pick one model — either add sphinx FKs (at minimum `sphinx_destination_whitelist.destination_id → sphinx_destinations`) or explicitly document the cleanup-cron as the design and ensure it covers every relationship, not just products.

### H5 — [High · Agent-reported] Critical write paths untested; integration harness structurally can't test writes
`submitOrder`, `fn_*_pre_place_order` (cart price correction), and `fn_travel_core_update_cscart_currencies` have **no tests**. `tests/Integration/BookingFlowTest.php` is mislabeled — it fakes the prices it "aggregates" and never touches a DB; `BookingRepositoryIntegrationTest` only exercises **reads** because `create/update`'s own transactions break test isolation. I.e. the C1 nesting defect is *why* the write paths are untestable.
*Fix:* fix C1, then add DB-backed tests for `create/update/delete` + the `travel_bookings` mirror, a stubbed-API `submitOrder` rollback/commit test, and a `pre_place_order` fixture test.

### M1 — [Medium · Agent-reported] `@`-suppressed, non-atomic schema migrations
`novoton_holidays/functions/install.php:529-550` runs a 7-step `TIMESTAMP→INT` migration on `novoton_cache` as separate `@db_query()` calls with no per-step guard; a mid-way failure (e.g. `DROP` succeeds, `CHANGE` fails) leaves the table broken and **silent** (errors suppressed). Same broad `@`-suppression across feature-type/facility migrations (:464-527).
*Fix:* guard each DDL step on current column state, drop `@` in favor of checked execution + `fn_log_event`; for disposable cache data prefer `DROP TABLE + recreate`.

### M2 — [Medium · Agent-reported] Exchange-rate currency update not transactional (money risk)
`travel_core/functions/exchange_rates.php:~179-247` loops one `UPDATE ?:currencies` per currency with no transaction; a mid-loop death leaves mixed old/new coefficients → inconsistent customer-facing pricing until the next cron.
*Fix:* wrap the loop in one transaction; make the verify-readback trigger a rollback, not just a log warning.

### M3 — [Medium · Agent-reported, softened] Dangling `product_id` on hotel rows
`novoton_hotels.product_id` / `sphinx_hotels.product_id` are plain indexed columns with no reverse FK; an admin-side product delete leaves stale links. *Partially mitigated:* novoton registers a `delete_product_post` hook and sphinx has `CleanupCommand` — **verify these actually null the pointers on both sides**; close any gap (hook to `SET product_id = NULL`, or FK `ON DELETE SET NULL`).

### M4 — [Medium · Agent-reported] Uninstall product-deletion + DROP ordering are fragile
`novoton install.php:61-75` loops `fn_delete_product()` with no batch/transaction/error-handling (partial deletes + timeout on large catalogs, right before tables are dropped). Uninstall DROPs rely on manual child-first ordering rather than `SET FOREIGN_KEY_CHECKS=0`.
*Fix:* batch + guard the product deletion; bracket the DROP sequence with `FOREIGN_KEY_CHECKS`.

### M5 — [Medium · Carried from 06-24, re-confirmed] CI / test-gating gaps
`fgo_invoicing` & `smart-bill` have **no PHPUnit jobs** in `ci.yml`; the integration suite is gated off (`INTEGRATION_TESTDB_READY` unset, test-db image not published) so **no DB-backed tests run on PRs**; `travel_core`/`sphinx` run `--no-coverage` with no coverage gate; no React/E2E tests.
*Fix:* add the missing unit jobs, build+publish the test-db image and flip the gate, add a coverage floor on booking/pricing/cron files.

### M6 — [Medium · Carried from 06-24] Biggest procedural files get no lint/format gate
`php-cs-fixer` and the CI `PHP Lint` job exclude `controllers/`, `hooks/`, `functions/` — i.e. the largest files (`sphinx func.php` 1223, `travel_core functions/hotels.php` 927, `novoton functions/hotels.php` 822, `novoton_price_compare.php` 1103). PHPStan still types them, but dead imports/style drift accrue exactly where review is hardest.
*Fix:* second cs-fixer finder (≥ `no_unused_imports`) + extend the lint `find` to `controllers functions hooks`.

### L1 — [Low · Carried from 06-24] Fragile `nofilter` sanitization in novoton templates
`novoton_booking/search.tpl` selectively un-escapes API HTML via `escape:'html'|replace:'lt;pgt;':'<p>'|…` — a hand-rolled allow-list that's easy to break. Replace with a real allow-list purifier / Smarty modifier. Confirm the raw sink `{$booking_engine_html nofilter}` is 100% server-built.

### L2 — [Low · Carried from 06-24] GET-reachable admin mutations lack CSRF tokens
`novoton_admin.php` `run_cron`/`update_prices` and several `sync_*`/`recompute_*` actions mutate on GET (authed-admin only). Hardening, not an open hole. *Fix:* require POST (CS-Cart then CSRF-protects); add a permission gate to `run_cron`.

### L3 — [Low/Info · Pending] Duplication & possibly-stale dirs
novoton and sphinx carry parallel provider structures (booking/search/pricing/controllers) — a full "what belongs in `travel_core`" duplication map is **pending** (agent cut off). Also verify the `addon-smart-bill/` and `addon_fgo/` top-level dirs aren't stale duplicates of `fgo_invoicing`.

### L4 — [Info · VERIFIED] `LIBXML_PARSEHUGE` on the large-payload XML parse
`novoton_holidays/src/NovotonXmlParser.php:146` enables `LIBXML_PARSEHUGE` (removes libxml node depth/size limits) for large hotel-list responses. Not an XXE/injection issue (network + entities are off), but a very large or malformed upstream payload could drive high memory use. Bound it with an upstream response-size cap / memory guard if the Novoton endpoint is ever untrusted.
